## 二、待修复项清单

### 🔴 P0：安全相关（必须补齐）

| 序号 | 问题 | 涉及文档 | 修复要求 |
|------|------|----------|----------|
| A1 | **DELETE 接口鉴权约定模糊** | 04-API接口文档、01-系统架构 | 明确：① DELETE `/api/guestbook/[id]` 在上线前必须实现鉴权，否则不得对外暴露；② 临时方案采用 `Authorization: Bearer <ADMIN_API_KEY>`，环境变量 `ADMIN_API_KEY`；③ 生产方案可后续接入 NextAuth 等 |
| A2 | **URL 协议校验策略缺失** | 01-系统架构 | 在「5.1 安全」中新增条目：**外部内容 URL 白名单** — content JSON（作品集、社交链接）中的 `url` 字段在渲染前必须校验协议，仅允许 `http:`、`https:`、相对路径 `/`，禁止 `javascript:` 等；建议在 `lib/utils/` 提供 `isValidHref(url)` 供组件调用 |

### 🟡 P1：设计与约束缺失（建议补齐）

| 序号 | 问题 | 涉及文档 | 修复要求 |
|------|------|----------|----------|
| B1 | **限流存储策略未约定** | 01-系统架构 | 在「5.1 安全 → 表单防刷」中补充：若采用内存限流（非 Upstash），必须实现 **TTL 清理** 或 **LRU 上限**，防止 Map 无限增长导致 OOM；推荐优先使用 Upstash Rate Limit |
| B2 | **邮件发送失败处理未约定** | 01-系统架构、04-API接口文档 | ① 在 01「5.1 安全」或「数据流」中说明：联系表单提交后，若 `sendContactNotificationEmail` 失败，应记录日志并考虑返回 `{ success: true, data: { mailSent: false } }` 或 503；② 在 04「三、联系表单 API」的响应示例中补充 `mailSent` 字段说明 |
| B3 | **content 加载错误处理未约定** | 02-项目目录树、01-系统架构 | 在 02「content/ 职责」或 01 中补充：`lib/content/*.ts` 加载 JSON 失败时，需区分 `ENOENT`（文件不存在）与 `SyntaxError`（解析错误），至少 `console.error` 输出错误类型，便于排查；禁止空 catch 吞掉所有异常 |

## 三、文档修改指引

### 3.1 01-系统架构.md

**在「五、安全与性能考量 → 5.1 安全」表中新增：**

| 措施 | 说明 |
|------|------|
| 外部内容 URL 白名单 | content JSON 中的 url 仅允许 http/https/相对路径，渲染前校验 |
| 管理接口鉴权 | DELETE 等管理接口上线前必须鉴权（Bearer Token 或 NextAuth），否则不得暴露 |
| 限流存储策略 | 若用内存限流，须 TTL/LRU 防止 OOM；推荐 Upstash |
| 邮件失败处理 | 联系表单邮件发送失败时记录日志，可选返回 mailSent: false 或 503 |

**在「5.2 性能」或「六、扩展与维护」中新增：**

- 外部图片：`next.config.js` 的 `remotePatterns` 需配置作品集 thumbnail 等外部图源域名
